BUG-001 Spring Boot Version 3.3.0 No Longer Available CRITICAL RESOLVED

? What Happened

When downloading the project from Spring Initializr with bootVersion=3.3.0, the server returned a 400 Bad Request error: 

"Invalid Spring Boot version '3.3.0', Spring Boot compatibility range is >=3.5.0"

? Root Cause

Spring Initializr only supports currently maintained Spring Boot versions. Version 3.3.x has reached end-of-life and was removed from the Initializr.

? Resolution

Changed the Spring Boot version to 3.5.0 (latest stable):

<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.5.0</version>  <!-- was 3.3.0 -->
</parent>
? How I Found This: I used curl to download from Spring Initializr. The downloaded file was only 189 bytes — way too small for a zip. I checked the file contents with type command and saw the JSON error response instead of a zip archive. Lesson: Always check file size after download!
BUG-002 Cannot Load Driver Class: org.h2.Driver CRITICAL RESOLVED

? What Happened

Application crashed on startup with this error:

Cannot load driver class: org.h2.Driver
BeanCreationException: Error creating bean with name 'entityManagerFactory'
Failed to instantiate [com.zaxxer.hikari.HikariDataSource]

? Root Cause

Two problems combined:

  • Missing application.properties config — JPA was on the classpath but no datasource URL, username, or H2 settings were configured. Spring tried to auto-configure a datasource but couldn't find the driver.
  • Explicit driver-class-name set — Setting spring.datasource.driver-class-name=org.h2.Driver explicitly caused a classloader conflict with DevTools (which uses a separate restart classloader).

? Resolution

Added proper H2 configuration in application.properties and let Spring Boot auto-detect the driver from the JDBC URL (no explicit driver-class-name):

# H2 Database
spring.datasource.url=jdbc:h2:mem:issuedb;DB_CLOSE_ON_EXIT=FALSE
spring.datasource.username=sa
spring.datasource.password=

# H2 Console
spring.h2.console.enabled=true
spring.h2.console.path=/h2-console

# JPA
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
spring.jpa.open-in-view=false
? How I Found This: I read the full stack trace carefully. The key line was Cannot load driver class: org.h2.Driver. Then I checked application.properties — it only had spring.application.name, nothing else! Spring Boot auto-configures JPA when it's on the classpath, but it needs a datasource URL. Lesson: When using JPA, always configure your datasource properties!
BUG-003 Port 8080 Already In Use MAJOR RESOLVED

? What Happened

After fixing Bug #2, the app failed again with:

Web server failed to start. Port 8080 was already in use.
Identify and stop the process that's listening on port 8080
or configure this application to listen on another port.

? Root Cause

The previous failed run (Bug #2) left a zombie Java process still holding port 8080. Even though the app crashed, the JVM process didn't fully terminate.

? Resolution

Killed the orphan process using this command:

REM Find and kill the process on port 8080
for /f "tokens=5" %a in ('netstat -aon ^| findstr :8080 ^| findstr LISTENING') do taskkill /F /PID %a

Then restarted the app — it started successfully on port 8080.

? How I Found This: The error message was very clear: Port 8080 was already in use. I used netstat to find which process (PID) was using port 8080, then killed it with taskkill. Lesson: Always kill previous app instances before restarting! Or add DB_CLOSE_ON_EXIT=FALSE to your H2 URL to prevent shutdown race conditions.

? Key Takeaways (What You Should Learn)
  • Always check downloaded file sizes — A tiny file often means an error response, not actual content.
  • Read the full stack trace — The root cause is usually in the Caused by: sections at the bottom.
  • JPA requires datasource config — If spring-boot-starter-data-jpa is on the classpath, you MUST configure a datasource URL.
  • Let Spring auto-detect drivers — Don't set driver-class-name manually; Spring detects it from the JDBC URL.
  • Kill zombie processes — Use netstat -aon | findstr :PORT + taskkill /F /PID on Windows.
  • Use DB_CLOSE_ON_EXIT=FALSE in H2 URLs to prevent premature shutdown of in-memory databases.
  • Set spring.jpa.open-in-view=false to avoid the warning and prevent lazy-loading issues in controllers.
